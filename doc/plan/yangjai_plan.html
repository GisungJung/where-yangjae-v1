1. 프로젝트 개요

1.1 배경

양재역 주변 맛집 정보를 Google Sheets로 관리하던 팀 내 문화를 웹 서비스로 전환합니다. 기존 데이터(점심·저녁/회식 2개 시트, 약 80개 식당, 9명 평가자)를 그대로 마이그레이션하고, 누구나 익명(닉네임)으로 평가를 추가할 수 있는 환경을 만듭니다.

1.2 문제 정의

1
스프레드시트의 한계
모바일 가독성 떨어짐, 평가자 늘어날 때 열 추가 번거로움, 동시 편집 시 충돌
2
검색·필터 부재
"오늘 점심 한식 뭐 먹지?" 같은 일상 질문에 시트가 답을 주지 못함
3
위치 정보 단절
방향 정보만 있고 실제 지도와 연결 안 됨, 처음 가는 사람은 매번 검색
4
참여 진입장벽
시트 편집 권한, 서식 깨짐 우려로 평가 입력이 일부 인원에게 집중

2. 사용자 시나리오

2.1 페르소나

페르소나특징주요 행동
탐색가 · 매일 점심 새 메뉴 시도모바일 위주, 빠른 의사결정홈에서 평점순 정렬, 카테고리 필터, 룰렛 자주 활용
평가자 · 음식 좋아하고 의견 적극한줄평을 정성껏 작성먹은 식당 즉시 평가, 다른 사람 한줄평 비교 참고
관전자 · 평가는 안 하지만 정보는 본다로그인·평가 입력 부담스러움익명으로 둘러보기만, 평점 높은 곳 위주 탐색
큐레이터 · 새 식당 잘 발견맛집 등록을 적극적으로 함지도 검색으로 빠르게 식당 추가, 메뉴·비고 꼼꼼히 작성

2.2 핵심 시나리오 (User Journey)

시나리오 A — 11시 50분, 점심 메뉴 결정

홈 진입 → "오늘 뭐먹지?" 클릭 → 한식·일식 체크 → 룰렛 결과 확인 → 상세에서 지도 확인 → 출발

시나리오 B — 식사 후 평가 입력

상세 페이지 → "평가하기" 버튼 → 닉네임 선택 → ★ 4.0 + 한줄평 → 제출 → 상세로 복귀

시나리오 C — 새 식당 발견 후 등록

/add 진입 → "손수동" 검색 → 카카오 결과 선택 → 카테고리·메뉴 입력 → 등록 완료

6.6 뷰 (View) — 평점 집계 (정규화)

CREATE VIEW restaurant_stats AS
WITH latest_per_reviewer AS (
  SELECT DISTINCT ON (restaurant_id, reviewer_id)
    restaurant_id, reviewer_id, score
  FROM ratings
  ORDER BY restaurant_id, reviewer_id, created_at DESC
)
SELECT
  r.id,
  r.name,
  COUNT(l.score) AS rating_count,
  ROUND(AVG(l.score)::numeric, 1) AS avg_score
FROM restaurants r
LEFT JOIN latest_per_reviewer l ON l.restaurant_id = r.id
GROUP BY r.id, r.name;

reviewer_id별 최신 1건만 평균에 반영 — 같은 사용자의 N번째 평가는 "사실상 수정"으로 동작.

7.3 주의 사항

  • 휴업·폐업 식당 — status 필드(운영중/휴업/폐업)로 마킹, 레코드는 보존 (soft delete, 과거 평가 유지)
  • 좌표 보완 실패 — lat/lng = NULL 허용, 지도 영역만 미표시
  • 한줄평 없는 점수 — comment = NULL로 정상 삽입
  • 닉네임 중복 허용 — UNIQUE 없음. 같은 닉네임이 reviewer_id로 분리되어 중복 가능 (사칭 위험은 §16 리스크 표 참조).
  • 점수 형식 — "3.5", "4" 모두 numeric으로 변환, 빈 셀은 SKIP

카테고리(다중), 시트(점심/저녁), 휴업·폐업 포함 보기 토글
정렬
평점 높은 순(기본), 평가 많은 순, 이름 가나다 순
검색
식당명, 메뉴 실시간 검색 (debounce 300ms)
카드 정보
이름, 카테고리, 평균 평점, 평가 수, 한줄 비고

8.2 맛집 상세

  • 카카오맵 임베드 (260px 높이) + 위치 핀
  • 평균 평점 (큰 별점) + 평가 수 + 카테고리 배지
  • 평가자별 점수 막대그래프 (간단 시각화)
  • 한줄평 목록 — 닉네임 / 점수 / 코멘트 / 작성일
  • "평가하기" CTA 버튼
  • 네이버 지도 원본 링크

8.3 평가 입력

  • 닉네임 — 자유 입력 (1~20자). 같은 닉네임 중복 허용. localStorage의 reviewer_id로 본인 식별. 변경 시 본인 reviewers row UPDATE.
  • 별점 — 0.5점 단위, 터치/클릭으로 선택, 키보드 접근성
  • 한줄평 — textarea (200자 제한, 카운터 표시)
  • 사진 첨부 — 최대 3장, 클라이언트 1024px·JPEG 0.8 리사이즈 후 업로드. 첫 장 즉시·나머지 lazy.
  • 기존 본인 평가가 있어도 차단하지 않음 — "수정" 버튼 또는 새 평가 작성 둘 다 가능, 평균은 최신 1건만 반영.
  • 제출 버튼 — 로딩 상태, 성공 시 토스트 + 상세로 이동.

8.4 맛집 등록

컨셉: 구글시트 등록 방식 그대로
로그인 없이 닉네임만 선택하면 등록 가능. 누가 추가했는지 흔적만 남기는 가벼운 식별 (Google Sheets에서 행 추가하던 워크플로우의 디지털 버전).
  • 등록자 닉네임 (필수) — 자유 입력. localStorage의 reviewer_id로 본인 식별, reviewers 테이블 자동 upsert.
  • 장소 검색 — 카카오 키워드 검색, 결과 5개 카드 표시
  • 선택 시 — 이름·주소·좌표·kakao_place_id 자동 채움
  • 카테고리·시트 — 드롭다운 선택
  • 메뉴·비고 — textarea (선택 입력)
  • 중복 검사 — 같은 kakao_place_id 또는 이름 매칭 시 경고 banner
  • 등록 완료 시 — restaurants.registered_by에 닉네임 FK 저장

8.5 오늘 뭐먹지? (랜덤 추천)

UX 흐름
  1. 카테고리 멀티 체크 (선택 안 하면 전체)

8.6 운영 상태 관리 (휴업 · 폐업)

모델: status 필드 기반 soft delete

10. 보안 & RLS 정책

익명 시스템의 보안 핵심
Supabase RLS(Row Level Security)로 직접 SQL 주입을 막고, 클라이언트 키만 노출되어도 안전한 구조를 만듭니다. 추가로 어뷰징 방지 장치를 둡니다.

10.1 RLS 정책

테이블SELECTINSERTUPDATEDELETE
restaurants모두 허용모두 허용모두 허용차단 (Studio만)
reviewers모두 허용모두 허용본인만 (id ↔ x-reviewer-id 헤더)차단
ratings모두 허용모두 허용본인 reviewer_id 평가만본인 reviewer_id 평가만
rating_photos모두 허용본인 rating에만차단본인 reviewer_id 평가만

10.2 정책 SQL 예시

-- restaurants: 누구나 조회·생성 가능, 삭제 차단
ALTER TABLE restaurants ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Anyone can read restaurants"
  ON restaurants FOR SELECT USING (true);

CREATE POLICY "Anyone can insert restaurants"
  ON restaurants FOR INSERT WITH CHECK (true);

-- ratings: 본인이 등록한 평가만 수정 가능 (localStorage 닉네임 검증)
CREATE POLICY "Update own rating"
  ON ratings FOR UPDATE
  USING (reviewer_id::text = current_setting('request.headers', true)::json->>'x-reviewer-id');
